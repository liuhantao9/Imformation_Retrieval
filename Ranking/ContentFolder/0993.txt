	<h1 id="firstHeading" class="firstHeading" lang="en">Cryptographic protocol</h1>
	
	<div id="bodyContent" class="mw-body-content">
		<div id="siteSub" class="noprint">From Wikipedia, the free encyclopedia</div>
		<div id="contentSub"><span class="mw-redirectedfrom">&#160;&#160;(Redirected from <a href="/w/index.php?title=Security_protocol&amp;redirect=no" class="mw-redirect" title="Security protocol">Security protocol</a>)</span></div>
		
		
		
		<div id="jump-to-nav"></div>
		<a class="mw-jump-link" href="#mw-head">Jump to navigation</a>
		<a class="mw-jump-link" href="#p-search">Jump to search</a>
		<div id="mw-content-text" lang="en" dir="ltr" class="mw-content-ltr"><div class="mw-parser-output"><p>A <b>security protocol</b> (<b>cryptographic protocol</b> or <b>encryption protocol</b>) is an abstract or concrete <a href="/wiki/Communications_protocol" class="mw-redirect" title="Communications protocol">protocol</a> that performs a <a href="/wiki/Information_security" title="Information security">security</a>-related function and applies <a href="/wiki/Cryptographic" class="mw-redirect" title="Cryptographic">cryptographic</a> methods, often as sequences of <a href="/wiki/Cryptographic_primitive" title="Cryptographic primitive">cryptographic primitives</a>. A protocol describes how the <a href="/wiki/Algorithm" title="Algorithm">algorithms</a> should be used. A sufficiently detailed protocol includes details about data structures and representations, at which point it can be used to implement multiple, interoperable versions of a program.<sup id="cite_ref-1" class="reference"><a href="#cite_note-1">&#91;1&#93;</a></sup>
</p><p>Cryptographic protocols are widely used for secure application-level data transport. A cryptographic protocol usually incorporates at least some of these aspects:
</p>
<ul><li><a href="/wiki/Key_agreement" class="mw-redirect" title="Key agreement">Key agreement</a> or establishment</li>
<li>Entity <a href="/wiki/Authentication" title="Authentication">authentication</a></li>
<li>Symmetric <a href="/wiki/Encryption" title="Encryption">encryption</a> and message authentication <a href="/wiki/Key_(cryptography)" title="Key (cryptography)"> </a> material construction</li>
<li>Secured application-level data transport</li>
<li><a href="/wiki/Non-repudiation" title="Non-repudiation">Non-repudiation</a> methods</li>
<li><a href="/wiki/Secret_sharing" title="Secret sharing">Secret sharing</a> methods</li>
<li><a href="/wiki/Secure_multi-party_computation" title="Secure multi-party computation">Secure multi-party computation</a></li></ul>
<p>For example, <a href="/wiki/Transport_Layer_Security" title="Transport Layer Security">Transport Layer Security</a> (TLS) is a cryptographic protocol that is used to secure web (<a href="/wiki/HTTPS" title="HTTPS">HTTPS</a>) connections. It has an entity authentication mechanism, based on the <a href="/wiki/X.509" title="X.509">X.509</a> system; a key setup phase, where a <a href="/wiki/Symmetric_encryption" class="mw-redirect" title="Symmetric encryption">symmetric encryption</a> key is formed by employing public-key cryptography; and an application-level data transport function. These three aspects have important interconnections. Standard TLS does not have non-repudiation support.
</p><p>There are other types of cryptographic protocols as well, and even the term itself has various readings; Cryptographic <i>application</i> protocols often use one or more underlying <a href="/wiki/Key_agreement" class="mw-redirect" title="Key agreement">key agreement methods</a>, which are also sometimes themselves referred to as "cryptographic protocols". For instance, TLS employs what is known as the <a href="/wiki/Diffie%E2%80%93Hellman_key_exchange" title="Diffie–Hellman key exchange">Diffie–Hellman key exchange</a>, which although it is only a part of TLS <i>per se</i>, Diffie–Hellman may be seen as a complete cryptographic protocol in itself for other applications.
</p>

<h2><span class="mw-headline" id="Advanced_cryptographic_protocols">Advanced cryptographic protocols</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Cryptographic_protocol&amp;action=edit&amp;section=1" title="Edit section: Advanced cryptographic protocols">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>A wide variety of cryptographic protocols go beyond the traditional goals of data confidentiality, integrity, and authentication to also secure a variety of other desired characteristics of computer-mediated collaboration.<sup id="cite_ref-2" class="reference"><a href="#cite_note-2">&#91;2&#93;</a></sup> <a href="/wiki/Blind_signature" title="Blind signature">Blind signatures</a> can be used for <a href="/wiki/Ecash" title="Ecash">digital cash</a> and <a href="/wiki/Digital_credential" title="Digital credential">digital credentials</a> to prove that a person holds an attribute or right without revealing that person's identity or the identities of parties that person transacted with.  <a href="/wiki/Trusted_timestamping" title="Trusted timestamping">Secure digital timestamping</a> can be used to prove that data (even if confidential) existed at a certain time.  <a href="/wiki/Secure_multiparty_computation" class="mw-redirect" title="Secure multiparty computation">Secure multiparty computation</a> can be used to compute answers (such as determining the highest bid in an auction) based on confidential data (such as private bids), so that when the protocol is complete the participants know only their own input and the answer. <a href="/wiki/End-to-end_auditable_voting_systems" title="End-to-end auditable voting systems">End-to-end auditable voting systems</a> provide sets of desirable privacy and auditability properties for conducting <a href="/wiki/E-voting" class="mw-redirect" title="E-voting">e-voting</a>.  <a href="/wiki/Undeniable_signature" title="Undeniable signature">Undeniable signatures</a> include interactive protocols that allow the signer to prove a forgery and limit who can verify the signature.  <a href="/wiki/Deniable_encryption" title="Deniable encryption">Deniable encryption</a> augments standard encryption by making it impossible for an attacker to mathematically prove the existence of a plain text message. <a href="/w/index.php?title=Anonymous_re-mailer&amp;action=edit&amp;redlink=1" class="new" title="Anonymous re-mailer (page does not exist)">Digital mixes</a> create hard-to-trace communications.
</p>
<h2><span class="mw-headline" id="Formal_verification">Formal verification</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Cryptographic_protocol&amp;action=edit&amp;section=2" title="Edit section: Formal verification">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>Cryptographic protocols can sometimes be <a href="/wiki/Formal_verification" title="Formal verification">verified formally</a> on an abstract level. When it is done, there is a necessity to formalize the environment in which the protocol operates in order to identify threats. This is frequently done through the <a href="/wiki/Dolev-Yao" class="mw-redirect" title="Dolev-Yao">Dolev-Yao</a> model.
