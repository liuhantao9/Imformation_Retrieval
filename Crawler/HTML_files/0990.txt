	<h1 id="firstHeading" class="firstHeading" lang="en">Burrows–Abadi–Needham logic</h1>
	
	<div id="bodyContent" class="mw-body-content">
		<div id="siteSub" class="noprint">From Wikipedia, the free encyclopedia</div>
		<div id="contentSub"><span class="mw-redirectedfrom">&#160;&#160;(Redirected from <a href="/w/index.php?title=BAN_logic&amp;redirect=no" class="mw-redirect" title="BAN logic">BAN logic</a>)</span></div>
		
		
		
		<div id="jump-to-nav"></div>
		<a class="mw-jump-link" href="#mw-head">Jump to navigation</a>
		<a class="mw-jump-link" href="#p-search">Jump to search</a>
		<div id="mw-content-text" lang="en" dir="ltr" class="mw-content-ltr"><div class="mw-parser-output"><p><b>Burrows–Abadi–Needham logic</b> (also known as the <b>BAN logic</b>) is a set of rules for defining and analyzing information exchange protocols. Specifically, BAN logic helps its users determine whether exchanged information is trustworthy, secured against eavesdropping, or both. BAN logic starts with the assumption that all information exchanges happen on media vulnerable to tampering and public monitoring. This has evolved into the popular security mantra, "Don't trust the network."
</p><p>A typical BAN logic sequence includes three steps:<sup id="cite_ref-1" class="reference"><a href="#cite_note-1">&#91;1&#93;</a></sup>
</p>
<ol><li>Verification of message origin</li>
<li>Verification of message <a href="/wiki/Freshness_(cryptography)" class="mw-redirect" title="Freshness (cryptography)">freshness</a></li>
<li>Verification of the origin's trustworthiness.</li></ol>
<p>BAN logic uses <a href="/wiki/Postulate" class="mw-redirect" title="Postulate">postulates</a> and <a href="/wiki/Definition" title="Definition">definitions</a> – like all <a href="/wiki/Axiomatic_system" title="Axiomatic system">axiomatic systems</a> – to analyze <a href="/wiki/Authentication" title="Authentication">authentication</a> <a href="/wiki/Protocol_(computing)" class="mw-redirect" title="Protocol (computing)">protocols</a>. Use of the BAN logic often accompanies a <a href="/wiki/Security_protocol_notation" title="Security protocol notation">security protocol notation</a> formulation of a protocol and is sometimes given in papers.
</p>

<h2><span class="mw-headline" id="Language_type">Language type</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Burrows%E2%80%93Abadi%E2%80%93Needham_logic&amp;action=edit&amp;section=1" title="Edit section: Language type">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>BAN logic, and logics in the same family, are <a href="/wiki/Decidable_language" class="mw-redirect" title="Decidable language">decidable</a>: there exists an algorithm taking BAN hypotheses and a purported conclusion, and that answers whether or not the conclusion is derivable from the hypotheses. The proposed algorithms use a variant of <a href="/wiki/Datalog" title="Datalog">magic sets</a>.<sup id="cite_ref-2" class="reference"><a href="#cite_note-2">&#91;2&#93;</a></sup>
</p>
<h2><span class="mw-headline" id="Alternatives_and_criticism">Alternatives and criticism</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Burrows%E2%80%93Abadi%E2%80%93Needham_logic&amp;action=edit&amp;section=2" title="Edit section: Alternatives and criticism">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>BAN logic inspired many other similar formalisms, such as <a href="/w/index.php?title=Gong%E2%80%93Needham%E2%80%93Yahalom_logic&amp;action=edit&amp;redlink=1" class="new" title="Gong–Needham–Yahalom logic (page does not exist)">GNY logic</a>. Some of these try to repair one weakness of BAN logic: the lack of a good semantics with a clear meaning in terms of knowledge and possible universes. However, starting in the mid-1990s, crypto protocols were analyzed in operational models (assuming perfect cryptography) using model checkers, and numerous bugs were found in protocols that were "verified" with BAN logic and related formalisms.<sup class="noprint Inline-Template Template-Fact" style="white-space:nowrap;">&#91;<i><a href="/wiki/Wikipedia:Citation_needed" title="Wikipedia:Citation needed"><span title="What bugs? Which protocols? (November 2015)">citation needed</span></a></i>&#93;</sup> In some cases a protocol was reasoned as secure by the BAN analysis but where in fact insecure.<sup id="cite_ref-3" class="reference"><a href="#cite_note-3">&#91;3&#93;</a></sup> This has led to the abandonment of BAN-family logics in favor of proof methods based on standard invariance reasoning.<sup class="noprint Inline-Template Template-Fact" style="white-space:nowrap;">&#91;<i><a href="/wiki/Wikipedia:Citation_needed" title="Wikipedia:Citation needed"><span title="This claim needs references to reliable sources. (January 2017)">citation needed</span></a></i>&#93;</sup>
</p>
<h2><span class="mw-headline" id="Basic_rules">Basic rules</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Burrows%E2%80%93Abadi%E2%80%93Needham_logic&amp;action=edit&amp;section=3" title="Edit section: Basic rules">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>The definitions and their implications are below (<i>P</i> and <i>Q</i> are network agents, <i>X</i> is a message,
and <i>K</i> is an <a href="/wiki/Encryption_key" class="mw-redirect" title="Encryption key">encryption key</a>):
</p>
<ul><li><i>P</i> believes <i>X</i>: <i>P</i> acts as if <i>X</i> is true, and may assert <i>X</i> in other messages.</li>
<li><i>P</i> has jurisdiction over <i>X</i>: <i>P</i>'s beliefs about <i>X</i> should be trusted.</li>
<li><i>P</i> said <i>X</i>: At one time, <i>P</i> transmitted (and believed) message <i>X</i>, although <i>P</i> might no longer believe <i>X</i>.</li>
<li><i>P</i> sees <i>X</i>: <i>P</i> receives message <i>X</i>, and can read and repeat <i>X</i>.</li>
<li>{<i>X</i>}<sub><i>K</i></sub>: <i>X</i> is encrypted with key <i>K</i>.</li>
<li>fresh(<i>X</i>): <i>X</i> has not previously been sent in any message.</li>
<li>key(<i>K</i>, <i>P</i>↔<i>Q</i>): <i>P</i> and <i>Q</i> may communicate with shared key <i>K</i></li></ul>
<p>The meaning of these definitions is captured in a series of postulates:
</p>
<ul><li>If <i>P</i> believes key(<i>K</i>, <i>P</i>↔<i>Q</i>), and <i>P</i> sees {<i>X</i>}<sub><i>K</i></sub>, then <i>P</i> believes (<i>Q</i> said <i>X</i>)</li>
<li>If <i>P</i> believes (<i>Q</i> said <i>X</i>) and <i>P</i> believes fresh(<i>X</i>), then <i>P</i> believes (<i>Q</i> believes <i>X</i>).</li></ul>
<p><i>P</i> must believe that <i>X</i> is fresh here.  If <i>X</i> is not known to be fresh, then it might be an obsolete message, replayed by an attacker.
</p>
<ul><li>If <i>P</i> believes (<i>Q</i> has jurisdiction over <i>X</i>) and <i>P</i> believes (<i>Q</i> believes <i>X</i>), then <i>P</i> believes <i>X</i></li>
<li>There are several other technical postulates having to do with composition of messages.  For example, if <i>P</i> believes that <i>Q</i> said &lt;<i>X</i>, <i>Y</i>&gt;, the concatenation of <i>X</i> and <i>Y</i>, then <i>P</i> also believes that <i>Q</i> said <i>X</i>, and <i>P</i> also believes that <i>Q</i> said <i>Y</i>.</li></ul>
<p>Using this notation, the assumptions behind an authentication protocol
can be formalized.  Using the postulates, one can prove that certain
agents believe that they can communicate using certain keys.  If the
proof fails, the point of failure usually suggests an attack which
compromises the protocol.
</p>
<h2><span class="mw-headline" id="BAN_logic_analysis_of_the_Wide_Mouth_Frog_protocol">BAN logic analysis of the Wide Mouth Frog protocol</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Burrows%E2%80%93Abadi%E2%80%93Needham_logic&amp;action=edit&amp;section=4" title="Edit section: BAN logic analysis of the Wide Mouth Frog protocol">edit</a><span class="mw-editsection-bracket">]</span></span></h2>

<p>Agents A and B are equipped with keys <i>K</i><sub><i>as</i></sub> and <i>K</i><sub><i>bs</i></sub>, respectively, for communicating securely with S.  So we have assumptions:
</p>
<dl><dd><i>A</i> believes key(<i>K</i><sub><i>as</i></sub>, <i>A</i>↔<i>S</i>)</dd>
<dd><i>S</i> believes key(<i>K</i><sub><i>as</i></sub>, <i>A</i>↔<i>S</i>)</dd>
<dd><i>B</i> believes key(<i>K</i><sub><i>bs</i></sub>, <i>B</i>↔<i>S</i>)</dd>
<dd><i>S</i> believes key(<i>K</i><sub><i>bs</i></sub>, <i>B</i>↔<i>S</i>)</dd></dl>
<p>Agent <i>A</i> wants to initiate a secure conversation with <i>B</i>.  It therefore invents a key, <i>K</i><sub><i>ab</i></sub>, which it will use to communicate with <i>B</i>.  <i>A</i> believes that this key is secure, since it made up the key itself:
</p>
<dl><dd><i>A</i> believes key(<i>K</i><sub><i>ab</i></sub>, <i>A</i>↔<i>B</i>)</dd></dl>
<p><i>B</i> is willing to accept this key, as long as it is sure that it came from <i>A</i>:
</p>
<dl><dd><i>B</i> believes (<i>A</i> has jurisdiction over key(<i>K</i>, <i>A</i>↔<i>B</i>))</dd></dl>
<p>Moreover, <i>B</i> is willing to trust <i>S</i> to accurately relay keys from <i>A</i>:
</p>
<dl><dd><i>B</i> believes (<i>S</i> has jurisdiction over (<i>A</i> believes key(<i>K</i>, <i>A</i>↔<i>B</i>)))</dd></dl>
<p>That is, if <i>B</i> believes that <i>S</i> believes that <i>A</i> wants to use a particular key to communicate with <i>B</i>, then <i>B</i> will trust <i>S</i> and believe it also.
</p><p>The goal is to have
</p>
<dl><dd><i>B</i> believes key(<i>K</i><sub><i>ab</i></sub>, <i>A</i>↔<i>B</i>)</dd></dl>
<p><i>A</i> reads the clock, obtaining the current time <i>t</i>, and sends the following message:
</p>
<dl><dd>1  <i>A</i>→<i>S</i>:  {<i>t</i>, key(<i>K</i><sub><i>ab</i></sub>, <i>A</i>↔<i>B</i>)}<sub><i>K<sub>as</sub></i></sub></dd></dl>
<p>That is, it sends its chosen session key and the current time to <i>S</i>, encrypted with its private authentication server key <i>K</i><sub><i>as</i></sub>.
</p><p>Since <i>S</i> believes that key(<i>K</i><sub><i>as</i></sub>, <i>A</i>↔<i>S</i>), and <i>S</i> sees {<i>t</i>, key(<i>K</i><sub><i>ab</i></sub>, <i>A</i>↔<i>B</i>)}<sub><i>K</i><sub><i>as</i></sub></sub>, 
then <i>S</i> concludes that <i>A</i> actually said {<i>t</i>, key(<i>K</i><sub><i>ab</i></sub>, <i>A</i>↔<i>B</i>)}.  (In particular, <i>S</i> believes that the message was not manufactured out of whole cloth by some attacker.)
</p><p>Since the clocks are synchronized, we can assume
</p>
<dl><dd><i>S</i> believes fresh(<i>t</i>)</dd></dl>
<p>Since <i>S</i> believes fresh(<i>t</i>) and <i>S</i> believes <i>A</i> said {<i>t</i>, key(<i>K</i><sub><i>ab</i></sub>, <i>A</i>↔<i>B</i>)}, 
<i>S</i> believes that <i>A</i> actually <i>believes</i> that key(<i>K</i><sub><i>ab</i></sub>, <i>A</i>↔<i>B</i>).  (In particular, <i>S</i> believes that the message was not replayed by some attacker who captured it at some time in the past.)
</p><p><i>S</i> then forwards the key to <i>B</i>:
</p>
<dl><dd>2  <i>S</i>→<i>B</i>: {<i>t</i>, <i>A</i>, <i>A</i> believes key(<i>K</i><sub><i>ab</i></sub>, <i>A</i>↔<i>B</i>)}<sub><i>K<sub>bs</sub></i></sub></dd></dl>
<p>Because message 2 is encrypted with <i>K</i><sub><i>bs</i></sub>, and <i>B</i>
believes key(<i>K</i><sub><i>bs</i></sub>, <i>B</i>↔<i>S</i>), <i>B</i> now believes that <i>S</i>
said {<i>t</i>, <i>A</i>, <i>A</i> believes key(<i>K</i><sub><i>ab</i></sub>, <i>A</i>↔<i>B</i>)}.
Because the clocks are synchronized, <i>B</i> believes fresh(<i>t</i>), and so
fresh(<i>A</i> believes key(<i>K</i><sub><i>ab</i></sub>, <i>A</i>↔<i>B</i>)).  Because <i>B</i>
believes that <i>S</i>'s statement is fresh, <i>B</i> believes that <i>S</i> believes that
(<i>A</i> believes key(<i>K</i><sub><i>ab</i></sub>, <i>A</i>↔<i>B</i>)).  Because <i>B</i>
believes that <i>S</i> is authoritative about what <i>A</i> believes, <i>B</i> believes
that (<i>A</i> believes key(<i>K</i><sub><i>ab</i></sub>, <i>A</i>↔<i>B</i>)).  Because <i>B</i>
believes that <i>A</i> is authoritative about session keys between <i>A</i> and <i>B</i>, <i>B</i>
believes key(<i>K</i><sub><i>ab</i></sub>, <i>A</i>↔<i>B</i>).  <i>B</i> can now contact <i>A</i>
directly, using <i>K</i><sub><i>ab</i></sub> as a secret session key.
</p><p>Now let's suppose that we abandon the assumption that the clocks are
synchronized.  In that case, <i>S</i> gets message 1 from <i>A</i> with {<i>t</i>,
key(<i>K</i><sub><i>ab</i></sub>, <i>A</i>↔<i>B</i>)}, but it can no longer conclude
that t is fresh.  It knows that <i>A</i> sent this message at <i>some</i> time
in the past (because it is encrypted with <i>K</i><sub><i>as</i></sub>)
but not that this is a recent message, so <i>S</i> doesn't believe that <i>A</i>
necessarily wants to continue to use the key
<i>K</i><sub><i>ab</i></sub>.  This points directly at an attack on the
protocol: An attacker who can capture messages can guess one of the
old session keys <i>K</i><sub><i>ab</i></sub>.  (This might take a long
time.)  The attacker then replays the old {<i>t</i>,
key(<i>K</i><sub><i>ab</i></sub>, <i>A</i>↔<i>B</i>)} message, sending it to <i>S</i>.  If
the clocks aren't synchronized (perhaps as part of the same attack), <i>S</i> might
believe this old message and request that <i>B</i> use the old, compromised key
over again.
</p><p>The original <i>Logic of Authentication</i> paper (linked below) contains this example and many others, including analyses of the <a href="/wiki/Kerberos_(protocol)" title="Kerberos (protocol)">Kerberos</a> handshake protocol, and two versions of the <a href="/wiki/Andrew_Project" title="Andrew Project">Andrew Project</a> RPC handshake (one of which is defective).
</p>
